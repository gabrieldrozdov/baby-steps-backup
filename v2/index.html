Setting up your computer

1 Download a text editor

Click here to download Visual Studio Code (VS Code)!

[For Mac users] Once you’ve downloaded Visual Studio Code, make sure to drag it from your Downloads folder into your Applications folder.

Code is just text saved in a file. Unlike normal text files, code files are JUST text — they don’t contain any “styling” information like the font, color, or layout you’re using.

We have special apps called text editors that let us write code without any of that styling information. Text editors also give us many extra features to make writing code easier. The most important feature is called “syntax highlighting,” which dynamically colors your code to tell you if you’ve made any mistakes in your file. There are hundreds of programming languages each with their own rules, so this feature is an essential tool for analyzing code.

There are many text editors, but I like using Visual Studio Code (aka VS Code) because it’s popular, has a large community, and covers a wide range of features.

2 Create a GitHub account

Click here to sign up for a GitHub account!

[For college students] Use your personal email, not your school email, for your account. You may lose access to your school email, but you’ll want to keep your GitHub account for many years.

There are millions of developers working around the world. Whether they work alone or at a company, they need somewhere to reliably store their code. There are a few services that solve this problem, but GitHub is by and far the most popular one.

GitHub and similar services give developers a way to collaborate on the same codebase, save versions of code, and distribute their work. Its name comes from Git, which is the underlying tool that lets developers save versions of their code (although GitHub is one of many services that makes use of Git).

Microsoft currently owns and maintains both GitHub and Visual Studio Code. Because of this, there are many built-in connections between the two programs, which is a major reason why both applications are some of the most popular developer tools today.

Making your first website

5 Create and publish a new GitHub repository

GitHub repositories are folders that store your project’s code and other assets (like images). You can either create a repository via github.com or GitHub Desktop. We’re going to use GitHub desktop to do this. Here’s how:

5.1
In GitHub desktop, open the menu in the top-left corner.
5.2
In that menu, click Add and then Create New Repository...
5.3
Give your repository a name. Let’s name this first repository something like “my-first-website” (all lowercase, no spaces or punctuation except for hyphens). Once you’ve done that, click Create Repository (note that this name will soon be public on github.com so don’t write anything private here!).
5.4
You have just created a new folder in your computer’s Documents folder. We’re going to upload the contents of that folder to github.com. Click Publish repository. In the pop-up that appears, make sure Keep this code private ISN’T selected. Then, click Publish Repository.
5.5
Your repository is now live on github.com! Access it via GitHub Desktop by clicking View on GitHub or pressing COMMAND+SHIFT+G (Windows: CTRL+SHIFT+G). You can also get to the online repository via your profile on github.com directly.

A repository (or “repo”) is really just another word for a folder. We’re using repositories to store all of the files for our websites. Repositories are also used for tons of other things, including the codebases for much larger projects like social media sites or non-web projects like computer software.

6 Create an “index.html” file in your repository

6.1
With Visual Studio Code open, go to File > Open Folder. Then, you want to open the repository you just made. This should be located in [your-computer-username] > Documents > GitHub. Alternatively, you can just drag your repository’s folder into Visual Studio Code directly.
6.2
Let’s create a new file. Press COMMAND+N (Windows: CTRL + N) to create an unnamed new file in your repository. Alternatively, you can go to File > New Text File.
6.3
We want to save your file as “index.html”. Press COMMAND+SHIFT+S (Windows: CTRL+SHIFT+S) to “Save As”. Then, make sure you’re saving into your repository and name your file “index.html”. Again, this is possible to do via File > Save As too.

8 Test your code in a web browser

The quickest way to test your code is to drag your index.html into an open browser window.

If you edit your code, you will need to save your changes in your text editor and then refresh your browser to preview your changes.

[For Visual Studio Code users] Download the Live Server extension. This is a more accurate way of testing your code in a web browser. Once the extension is installed, you can set up an in-browser preview that automatically updates every time you save your code. Just follow these steps:

7.1
If you haven’t already, drag your repository into Visual Studio Code to open it.
7.2
Bring up the index.html file.
7.3
Press the Go Live button on the bottom of your screen or type COMMAND+L+O (Windows: ALT+L+O). Your browser should automatically open up a preview of your code.
7.4
If you close your browser tab and want to preview your code again, you can press COMMAND+L+O (Windows: ALT+L+O) at any time in Visual Studio Code to reopen the preview.

9 Commit and push your code

Every time you want to make changes to your repository public, you need to do two things: commit your changes, and then publish them.

8.1
In GitHub Desktop, your changes will appear in the panel on the left. (Make sure your repository is selected from the top-left menu if you’re not seeing this!)
8.2
On the bottom-left, you’ll see a few empty text fields. Every time you commit, you need to give your commit a name (the Summary) to help you keep track of what you actually changed in your code. The “Description” field is optional. Once you’ve done that, press Commit to main.
8.3
In the top-right corner, press Publish. This takes the code on your computer and sends it to github.com.
8.4
That’s it! Now, if you go to your repository’s page on github.com (COMMAND+SHIFT+G or CTRL+SHIFT+G), you’ll see your files on the internet.

When you commit changes to your code, you are saving a version of that code. If you need to, you are able to revert or refer to that older version to fix any issues that may pop up later on. That’s why every commit requires a name and has an option for a description — so that you can keep track of what you were actually changing.

Publishing code means making your code live on GitHub, and in many cases live on the internet. You can commit numerous times in a row to keep track of your work and then only publish when you’re ready to make your work publicly available online.

10 Deploy your repository GitHub pages

This is the last step to make your site live!

9.1
Open your repository on github.com. You can get to it via GitHub desktop or the sidebar on github.com (click your profile image in the top-right bar and then click “Your repositories”).
9.2
Go to its “Settings” page via the top navigation.
9.3
In the sidebar, click “Pages”.
9.4
In the “Branch” section, change the “None” dropdown to “main”.
9.5
That’s it! In a few moments (usually less than a minute), your site will be live. Once it’s live, you can refresh the page to find the live URL, or navigate to it directly at [your-username].github.io/my-first-website/ (or whatever your repository name is).

[If you accidentally set your repository to private in a previous step] Your repository has to be public to work on GitHub Pages. Follow these steps to change your repository’s visibility settings:

9.a
Go to your repository’s “Settings” page, go to the “General” menu in the sidebar.
9.b
Scroll all the way to the bottom to find the “Change repository visibility” button.
9.c
Follow all the prompts to change your repository’s visibility to public.

A website is just a series of files, just like a GitHub repository. To make a website live, we need to find a way to give those files to people. To do that, a physical computer somewhere in the world needs to host your files. Then, when a user accesses a specific URL, that computer will send those files to that user.

That’s the most basic kind of website. We call this ”static” web hosting because the user isn’t able to manipulate your code — they can only access and view it. More complicated websites, like social media sites, allow users to make permanent changes to the hosted files, like creating user accounts or uploading profile pictures.

While you have to pay to host more complicated sites, there are many free services that can host static sites. GitHub Pages is one such static web hosting service, where you can set any of your repositories to be available as a live website under GitHub’s .io URL structure. That’s what we just did by publishing our repository on GitHub Pages!

Why does GitHub offer this service for free even though we’re using their resources? Here are a few ideas. Since they’re already hosting your repository, it is likely very cheap for them to host the same files as a website. Plus, having an easy-to-use connection between GitHub repositories and static sites is a good incentive for developers to use GitHub for launching new projects. As you keep on coding, you’ll frequently run into web and non-web projects with an affiliated GitHub Pages site for documentation and marketing purposes.

Updating your website

11 Create new pages on your website

To create new pages on your website, you’ll need to create more .html files. While you can name your .html files whatever you want, it’s best to organize your repository into folders. Each folder represents a page on your site with it’s own unique .html file.

We do this for two reasons:

1
Having a separate folder for every page helps us organize our assets.
2
If you didn’t name your .html files “index.html”, you would end up with confusing and ugly URLs.

Here’s why that second reason matters. Let’s say your site is “gdwithgd.com”. If you want a page for all of your projects, you could create a file called “projects.html” and then a file for every project, like “project-1.html” and so on. But here are the URLs you’ll end up with:

gdwithgd.com (your “index.html” file)
gdiwthgd.com/projects.html
gdwithgd.com/project-1.html

In reality, you want your URL to look more like the following:

gdwithgd.com (your “index.html” file)
gdiwthgd.com/projects/ (the “index.html” file in the “project” subfolder)
gdwithgd.com/projects/project-1/ (the “index.html” file in the “project-1” subfolder of the “projects” subfolder)

This can be tricky to get the hang of! Below, you can find an example of the folder structure for a repository with several pages.

[your-repository-name]
index.html (your-site.com)
about
index.html (your-site.com/about/)
projects
index.html (your-site.com/projects/)
project-1
index.html (your-site.com/projects/project-1/)
project-2
index.html (your-site.com/projects/project-2/)
project-3
index.html (your-site.com/projects/project-3/)

Making your site more *professional*

14 Connect your repository to a custom URL

GitHub doesn’t offer URLs besides [your-username].github.io/[your-repository-name]. If you want a unique URL, you’ll have to purchase a license to one from something called a domain name registrar. There are many such services out there, but for the sake of this demo we’ll use Namecheap.

11.1
Create an account on namecheap.com. Use your personal email and not a school email.
11.2
Search for a domain you want to purchase. These can vary wildly in price, but most .com domains are below $20 per year.
11.3
Buy your domain (no need to purchase any of the extra add-ons advertised at checkout).

Now that you have your domain, let’s connect it to your GitHub repository!

11.4
Head to your account dashboard on namecheap.com (accessed via the “Account” dropdown in the navbar).
11.5
Find your domain name (your URL) in the list and click “Manage”.
11.6
Click on the “Advanced DNS” menu.
11.7
In the “Host Records” section of this page, you’ll find several records (like a “CNAME Record” or something of the sort). Delete all the existing entries.
11.7
We’re going to add several new records (as indicated by Namecheap’s tutorial):
Type: “A”, Host: “@”, Value: “185.199.108.153”
Type: “A”, Host: “@”, Value: “185.199.109.153”
Type: “A”, Host: “@”, Value: “185.199.110.153”
Type: “A”, Host: “@”, Value: “185.199.111.153”
Type: “CNAME”, Host: “www”, Value: “[your-github-username].github.io”
11.8
Now we’re going to switch over to github.com. Go to your repository’s page, click “Settings”, and then go to the “Pages” submenu from the sidebar.
11.9
In the “Custom domain” field, add the url of the domain name you are trying to connect (e.g. gdwithgd.com). Press “Save”.

And that’s it! After some time, your site will be accessible via the URL you purchased. This can be instant, or could take several hours. Try refreshing the “Pages” menu on GitHub every now and then to see the status of your page.

[Enforce HTTPS] Once your site is up-and-running, you’ll be given the option to check “Enforce HTTPS” on your “Pages” menu. You should do this! HTTPS is a security protocol for modern sites, and it’s bad to not have it activated.

[If you want a subdomain] Sometimes, you want multiple repositories hosted on the same site. The way I handle this is by using subdomains. One example of a subdomain is this site, babysteps.gdwithgd.com. I pay for gdwithgd.com, but can host an unlimited amount of repositories on the same URL using the format [subdomain].gdwithgd.com.

To do this, you just need to add one more record to your host records on namecheap.com:
Type: “CNAME”, Host: “[your-subdomain-name]”, Value: “[your-github-username].github.io”

The subdomain can be whatever you want (all lowercase, no punctuation except for hyphens).

Then on your repository’s page on github.com, add the URL in the same way you did before but formatted like a subdomain (e.g. babysteps.gdwithgd.com). And you’re done!

Technically speaking, you CAN own a URL. But to do that, you need to become a domain registrar, which means working with international organizations and paying a lot of fees. That is not feasible for basically anyone, so instead we work with DNS providers and pay them a much, much, MUCH smaller fee in exchange for licensing a URL.

Why are some URLs more expensive that others? Sometimes, DNS providers will jack up the prices of URLs that are either short or use common words or phrases. In other cases, someone owns the license for an attractive URL and is unwilling to part with it for less than a certain price. In more recent times, new top-level domains (everything after the dot, like .com or .org) have popped up that just start at higher prices than .com.

Fun fact: many unique top-level domains (like .io) are actually affiliated with specific countries. Map Men have a great video deep-dive on this topic.
